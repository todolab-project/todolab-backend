# ToDoLab Roadmap

ToDoLab은 단순히 할 일을 많이 쌓아두는 ToDo CRUD 앱이 아니라, 내가 오늘 실제로 끝낼 수 있는 크기로 일을 바꿔주는 실행 보조 앱으로 가져가고 싶다.

핵심은 할 일을 많이 관리하는 것이 아니라, 오늘 할 일을 적당한 크기로 고르고 끝내는 것이다.

```text
빠르게 기록
-> 오늘 할 일로 선별
-> 실행 가능한 크기로 쪼개기
-> 완료 처리
-> 완료 로그로 성취감 얻기
-> 미룬 일은 다시 정리하기
```

내가 중요하게 보고 싶은 기준은 이렇다.

- 기록은 최대한 빠르게 한다.
- 오늘 할 일은 너무 많이 쌓지 않는다.
- 큰 일은 작게 쪼갠다.
- 완료한 일은 눈에 보이게 남긴다.
- 계속 미루는 일은 그냥 방치하지 않고 다시 정리한다.

## 1. 지금 코드 상태

현재 코드는 일정 관리 앱에서 오늘 실행 중심 ToDo 앱으로 넘어가는 중이다.
기존 구조를 한 번에 갈아엎기보다는, 지금 구조 위에 1차 MVP에 필요한 상태 기반 ToDo 흐름을 얹는 방식으로 가는 게 좋겠다.

현재 구조를 보면:

- `Task`가 할 일/일정의 핵심 엔티티다.
- `TaskType`은 `SCHEDULE`, `TODO`, `IDEA`를 구분한다.
- `TaskRequest`, `TaskResponse`가 생성/수정 요청과 응답을 맡는다.
- `TaskController`는 `/api/tasks` 기반 CRUD와 날짜 범위 조회 API를 제공한다.
- `TaskService`는 생성, 조회, 수정, 삭제, 미정 항목 조회를 처리한다.
- `TaskTxService`는 수정 트랜잭션을 담당한다.
- `TaskRepositoryCustom`, `TaskRepositoryImpl`에서 Querydsl 기반 조회를 처리한다.
- `TaskPageController`는 `/tasks/inbox`, `/tasks/today`, `/tasks/log`, `/tasks/week`, `/tasks/month` 화면을 연결한다.
- `TaskViewService`는 주간/월간 화면에 필요한 모델을 만든다.
- 테스트는 controller, service, repository, view, batch 쪽으로 나뉘어 있다.

현재 특징:

- 날짜가 없는 항목은 장기적으로 `status == INBOX` 기준으로 다룬다.
- `TaskType`은 있지만, 화면과 조회 흐름은 아직 일정 기능과 ToDo 기능이 함께 남아 있다.
- `INBOX`, `TODAY`, `DONE` 상태 모델과 `targetDate`, `completedAt` 필드는 추가되어 있다.
- `Inbox -> Today -> Done -> Log` 기본 흐름은 API와 화면에 연결되어 있다.
- 하단 탭은 `Today / Calendar / D-Day / More`로 정리되어 있다.
- D-Day는 `DdayGoal` 별도 엔티티로 최소 등록, 조회, 삭제 흐름이 연결되어 있다.
- `schema.sql`을 직접 관리하고 있으므로 필드를 추가하면 스키마도 같이 바꿔야 한다.
- 1차 MVP의 기본 흐름은 닫혔고, 이제는 Today 중심 사용성을 다듬은 뒤 2차 고도화로 넘어가면 된다.

## 2. 1차 MVP 목표

1차 MVP에서는 모든 기능을 다 넣지 않는다.
먼저 아래 흐름이 자연스럽게 동작하는 것을 목표로 한다.

```text
할 일을 빠르게 등록한다
-> Inbox에 쌓인다
-> 오늘 할 일로 가져온다
-> 완료 처리한다
-> 완료 로그에 남는다
-> 못 한 일은 다음 날로 이월한다
```

1차 MVP에 포함할 기능:

- 빠른 기록함
- 오늘 할 일
- 완료 처리
- 완료 로그
- 미완료 이월

1차 MVP에서 아직 하지 않을 것:

- 큰 할 일 쪼개기
- 미룬 이유 선택
- 주간 리포트
- 자연어 입력
- AI 일정 제안

## 3. 데이터 모델 변경 방향

처음부터 모든 필드를 다 넣지는 않는다.
1차 MVP에는 실행 흐름에 꼭 필요한 필드만 추가한다.

우선 추가할 필드:

- `status`
  - `INBOX`
  - `TODAY`
  - `DONE`
- `targetDate`
  - 오늘 할 일로 지정된 날짜
- `completedAt`
  - 완료 시간

2차 이후로 미룰 필드:

- `carryOverCount`
- `parentId`
- `deferReason`
- `priority`

상태 의미:

- `INBOX`: 빠르게 기록했지만 아직 오늘 할 일로 고르지 않은 항목
- `TODAY`: 특정 날짜에 실행하기로 고른 항목
- `DONE`: 완료한 항목

기존 구조와의 관계:

- 1차 MVP에서는 `TaskType.TODO`를 기본 실행 대상 타입으로 본다.
- 기존 `SCHEDULE` 기반 Today/Week/Month 화면은 당장 제거하지 않는다.
- 날짜 없는 항목을 `unscheduled`로 판단하던 방식은 장기적으로 `status == INBOX` 기준으로 옮긴다.
- 기존 일정 기능과 새 ToDo 흐름이 충돌하지 않게 단계적으로 정리한다.

## 4. API 방향

기존 `/api/tasks` CRUD는 유지한다.
그 위에 상태 전이에 필요한 API만 작게 추가한다.

추가할 API 후보:

- `GET /api/tasks/inbox`
  - Inbox 목록 조회
- `GET /api/tasks/today?date=yyyy-MM-dd`
  - 특정 날짜의 Today 목록 조회
- `GET /api/tasks/done?date=yyyy-MM-dd`
  - 특정 날짜의 완료 로그 조회
- `PATCH /api/tasks/{id}/today`
  - Inbox 또는 다른 상태의 Task를 Today로 이동
- `PATCH /api/tasks/{id}/done`
  - Task 완료 처리
- `PATCH /api/tasks/{id}/carry-over`
  - 미완료 Today Task를 다음 날로 이월

생성 API는 이렇게 가져가고 싶다.

- 제목만으로 생성 가능해야 한다.
- 날짜가 없으면 `status = INBOX`로 저장한다.
- 오늘 할 일로 바로 만들 때는 `status = TODAY`, `targetDate = 오늘`로 저장할 수 있게 확장한다.

## 5. 화면 구성

1차 MVP에서는 Today 화면을 앱의 중심으로 둔다.
사용자가 앱을 켰을 때 바로 “오늘 뭘 하면 되는지” 보여야 한다.

화면 방향은 이렇게 가져가고 싶다.

- Today
- Calendar
- D-Day
- More

기록함과 완료 로그는 중요한 기능이지만, 하단 탭의 중심이 될 필요는 없을 수 있다.
기록함은 Today 안의 “아직 고르지 않은 일” 섹션으로 흡수하고, 완료 로그는 Today의 완료 요약과 Calendar/More에서 접근하는 방향을 검토한다.

하단 탭 후보:

- `Today`
- `Calendar`
- `D-Day`
- `More`

원칙:

- 첫 번째 탭은 무조건 Today로 둔다.
- Calendar는 일정/날짜 맥락을 보기 좋아서 하단에 남긴다.
- D-Day는 시험, 취업, 프로젝트 마감처럼 실제 사용 맥락이 강하므로 차별화 탭 후보로 둔다.
- More에는 완료 로그, 기존 Week/Month, 설정, 나중에 통계/계정 기능을 넣을 수 있다.

### Inbox

빠르게 등록한 할 일을 모아두는 기능이다.
별도 탭으로 계속 둘지, Today 안에 흡수할지는 다시 정리한다.

제품 방향상으로는 Today 안에 “아직 고르지 않은 일”로 노출하는 쪽이 더 자연스럽다.

필요한 기능:

- 제목만으로 빠른 등록
- 날짜 없는 할 일 목록
- Today로 이동
- 날짜 지정
- 삭제
- 카테고리 지정

### Today

앱의 메인 화면이다.

필요한 기능:

- 빠른 기록 입력창
- 오늘 할 일 목록
- 아직 고르지 않은 일 요약
- “며칠 전 기록” 같은 기록 시점 표시
- 오늘 완료한 일
- 오늘 완료 개수
- 연속 실행 요약
- 완료 체크
- 미완료 이월 처리

### Done Log

완료한 일을 날짜별로 보는 화면이다.
단독 하단 탭으로 둘 수도 있지만, 장기적으로는 Today 요약, Calendar 날짜 상세, More 안의 기록 화면으로 나눌 수 있다.

필요한 기능:

- 오늘 완료한 일 조회
- 날짜별 완료 목록 조회
- 완료 개수 표시
- 추후 주간 리포트로 확장

### Calendar

일정과 날짜별 흐름을 보는 화면이다.
기존 Week/Month 화면을 유지하되, ToDo 완료 로그나 D-Day와 연결할 수 있다.

필요한 기능:

- 주간/월간 일정 보기
- 날짜별 완료 로그 접근
- 날짜별 Today 항목 접근
- 나중에 D-Day 항목 표시

### D-Day

단순 카운트다운이 아니라, 목표와 오늘 실행을 연결하는 화면이다.

예시:

- `정보처리기사 D-12`
- 오늘 연결된 할 일: `기출 20문제 풀기`

필요한 기능:

- D-Day 항목 등록
- 목표 날짜 표시
- 남은 날짜 표시
- D-Day와 Today 할 일 연결
- 목표별 진행 할 일 목록

### More

하단 탭에 모두 담기 어려운 기능을 모아둔다.

후보:

- 완료 로그
- Week/Month Calendar 상세 링크
- 설정
- 카테고리 관리
- 나중에 통계, 계정, 알림 설정

## 6. 1차 구현 순서

### 6-1. 엔티티와 스키마 확장

먼저 Task가 상태 기반 ToDo 흐름을 표현할 수 있게 만든다.

해야 할 일:

- `TaskStatus` enum 추가
- `Task.status` 추가
- `Task.targetDate` 추가
- `Task.completedAt` 추가
- `schema.sql`에 `STATUS`, `TARGET_DATE`, `COMPLETED_AT` 추가
- 기존 데이터와 충돌하지 않도록 기본 상태 정책 정리

### 6-2. 도메인 상태 전이 메서드 추가

상태 변경은 Service에서 필드를 직접 만지기보다, `Task` 안에 의도가 드러나는 메서드로 둔다.

메서드 후보:

- `moveToInbox()`
- `moveToToday(LocalDate targetDate)`
- `complete(LocalDateTime completedAt)`
- `carryOverTo(LocalDate nextDate)`

원칙:

- Service는 흐름을 조합한다.
- 상태 전이 규칙은 가능하면 `Task` 도메인 메서드 안에 둔다.

### 6-3. Repository 조회 추가

필요한 조회:

- Inbox 목록 조회
- 날짜별 Today 목록 조회
- 날짜별 Done 목록 조회

정렬 기준:

- Inbox: 생성일 기준으로 통일
- Today: 우선 생성일 기준으로 시작
- Done Log: 완료 시간 최신순

### 6-4. Service 기능 추가

1차 MVP의 핵심 흐름을 Service에 추가한다.

필요한 기능:

- 빠른 등록 시 Inbox 저장
- Inbox 목록 조회
- Inbox에서 Today로 이동
- Today 목록 조회
- Today 할 일 완료 처리
- 완료 로그 조회
- 미완료 할 일 다음 날로 이월

### 6-5. Controller API 추가

기존 `TaskController`에 상태 기반 API를 추가한다.
처음부터 컨트롤러를 많이 나누지는 않는다.

나중에 기능이 커지면 `TaskWorkflowController` 같은 식으로 분리할 수 있다.

### 6-6. 화면 연결

API가 안정화된 뒤 화면을 붙인다.

우선순위:

- Today 화면을 메인으로 재정리
- Inbox 화면 추가 또는 기존 Unscheduled 화면을 Inbox로 전환
- Done Log 화면 추가
- 하단 내비게이션을 우선 `Inbox / Today / Log`로 정리
- 이후 제품 방향에 맞춰 `Today / Calendar / D-Day / More`로 재정리

### 6-7. Today 중심 재구성

현재는 Inbox와 Log가 별도 탭으로 나뉘어 있지만, 최종적으로는 Today를 더 강한 첫 화면으로 만든다.

해야 할 일:

- [x] 하단 첫 탭을 Today로 고정
- [x] 기록함을 Today 안의 섹션으로 흡수
- [x] Inbox 항목에 기록일 메타 표시
- [x] Today 안에서 Inbox 항목을 오늘 할 일로 이동
- [x] 오늘 완료 요약을 Today 하단에 유지
- [x] Log 탭은 More와 Calendar 날짜 상세에서 접근하도록 정리
- [x] 빠른 기록과 일반 생성의 타입 정책 정리

### 6-8. 하단 탭 구조 정리

다음 후보로 정리한다.

- [x] Today
- [x] Calendar
- [x] D-Day
- [x] More

D-Day는 별도 엔티티로 최소 등록, 조회, 삭제 화면부터 시작한다.
이후 Today 할 일 연결과 Calendar 표시를 붙여서 단순 카운트다운이 아니라 실행과 이어지게 만든다.

## 7. 테스트 계획

기존 테스트를 깨지지 않게 유지하면서, 1차 MVP 흐름 테스트를 추가한다.

최소 테스트 케이스:

- 제목만으로 Task 등록 가능
- 날짜 없는 Task는 Inbox 상태로 저장
- Inbox Task를 Today로 이동 가능
- Today Task 완료 시 `DONE` 상태와 `completedAt` 저장
- 완료 로그에서 완료된 Task 조회 가능
- 미완료 Today Task를 다음 날로 이월 가능

추가하면 좋은 테스트:

- 완료된 Task를 다시 완료 처리할 때의 정책
- DONE Task를 Today로 다시 이동할 수 있는지 정책
- `targetDate` 없는 TODAY 상태를 허용하지 않는지 검증
- 완료 로그 조회 시 날짜 범위가 정확한지 검증

## 8. 2차 고도화

1차 MVP가 안정화되면 ToDoLab만의 차별화 기능을 붙인다.

2차 범위:

- D-Day와 Today 할 일 연결
- 큰 할 일 쪼개기
- 미룬 일 감지
- 미룬 이유 선택
- 오늘 할 일 개수 제한
- 완료/연속 실행 요약
- 주간 완료 리포트

### D-Day와 실행 연결

D-Day는 단순히 날짜만 세는 기능으로 끝내지 않는다.
목표까지 남은 시간과 오늘 할 일을 연결해야 ToDoLab다운 기능이 된다.

해야 할 일:

- D-Day 엔티티는 `DdayGoal`로 분리한다.
- 목표 이름과 목표 날짜 저장
- D-Day에 연결된 Today 할 일 목록 표시
- D-Day 상세에서 오늘 할 일 추가
- Calendar에 D-Day 표시

예시:

- `정보처리기사 D-12`
- `포트폴리오 제출 D-5`
- `이직 지원 마감 D-3`

각 D-Day는 “오늘 가까워지기 위해 할 일”을 만들 수 있어야 한다.

### 큰 할 일 쪼개기

초기에는 AI 없이 직접 하위 작업을 추가하는 방식으로 간다.

해야 할 일:

- Task에 상위/하위 작업 관계 추가
- Parent Task와 Sub Task 관계 구성
- 하위 작업 완료율 표시
- 큰 작업은 하위 작업 기준으로 진행률 계산

### 미룬 일 관리

계속 미루는 일을 별도로 감지하고 다시 정리한다.

해야 할 일:

- 이월 횟수 기록
- 3번 이상 이월된 일은 미룬 일로 표시
- 미룬 이유 선택
- 미룬 일 액션 제공

미룬 이유 후보:

- 너무 큼
- 지금 필요 없음
- 하기 싫음
- 마감이 없음
- 다른 사람 대기 중

미룬 일 액션 후보:

- 작게 쪼개기
- 보류함으로 이동
- 날짜 재설정
- 삭제

### 오늘 할 일 개수 제한

오늘 할 일이 너무 많아지면 실행력이 떨어진다.
강제로 막기보다는, 사용자가 오늘 할 일이 너무 많다는 걸 인식하게 만든다.

해야 할 일:

- 오늘 할 일 권장 개수 설정
- 기본값은 3개 또는 5개로 시작
- 초과 등록 시 경고 또는 안내 표시
- 강제 제한보다는 피드백 중심으로 구현

### 완료와 연속 실행 요약

동기부여는 과한 게임화보다 조용한 성취감으로 가져간다.

보여줄 수 있는 정보:

- 오늘 완료한 일 개수
- 이번 주 완료한 일 개수
- 연속 실행 일수
- 어제보다 나아진 점

Today 화면에는 작은 요약으로 보여주고, 자세한 내용은 Log나 More에서 볼 수 있게 한다.

### 주간 완료 리포트

완료 로그를 기반으로 주간 리포트를 만든다.

해야 할 일:

- 이번 주 완료한 일 개수
- 카테고리별 완료 개수
- 가장 많이 처리한 카테고리
- 계속 미룬 일 목록
- 다음 주로 넘길 일 목록

## 9. 3차 차별화

기본 실행 흐름이 안정화된 뒤 앱을 더 매력적으로 만드는 기능을 붙인다.

후순위 기능:

- 아침 계획
- 저녁 회고
- 성취감 UI 고도화
- 자연어 입력
- AI 일정 제안

AI 기능은 처음부터 붙이지 않는다.
기록, Today, 완료 로그, 이월, 미룬 일 관리 흐름이 안정화된 뒤에 붙인다.

### 저녁 회고

밤에 앱을 열었을 때 하루를 정리할 수 있게 만든다.

보여줄 내용:

- 오늘 완료한 일
- 오늘 못 한 일
- 내일로 넘길 일
- 계속 미루는 일

저녁 회고는 완료 로그와 이월 데이터를 활용한다.
처음에는 자동 알림 없이 화면 안에서 자연스럽게 제공한다.

### AI 쪼개기

AI는 처음부터 핵심 기능으로 넣지 않는다.
큰 할 일 쪼개기 흐름이 직접 입력 방식으로 안정화된 뒤 붙인다.

예시:

- 사용자가 `이직 준비하기`를 입력한다.
- AI가 작은 실행 단위 후보를 제안한다.
- 사용자가 선택한 항목만 Today 또는 하위 작업으로 만든다.

## 10. Redis 도입 후보

Redis는 성능 개선이나 상태 관리가 실제로 필요해지는 지점에 붙인다.
처음부터 넣기보다는 병목이나 기능 요구가 생겼을 때 도입한다.

붙일 만한 위치:

- 세션 저장소: 로그인 이후 서버 세션 저장
- Today 조회 캐시: 사용자가 자주 여는 Today 화면 캐싱
- Done Log 집계 캐시: 주간 리포트 계산 결과 캐싱
- 자연어 파싱 결과 임시 저장: 사용자가 확정하기 전 임시 상태 보관
- rate limit: 로그인, 자연어 입력, 반복 생성 API 보호

우선순위:

- 로그인 이후 세션 저장소로 먼저 도입
- 조회/집계 캐시는 실제 성능 병목을 확인한 뒤 적용

## 11. Kafka 도입 후보

Kafka는 이벤트 흐름이 생긴 뒤에 붙인다.
단순 CRUD 단계에서 먼저 넣기보다는, 알림이나 비동기 처리가 필요해지는 시점에 도입한다.

붙일 만한 위치:

- Task 생성/완료/이월 이벤트 발행
- 완료 로그 기반 주간 리포트 생성
- 일정 공유 알림 이벤트 처리
- 자연어 일정 등록 요청 비동기 처리
- 일일 요약 메일 또는 저녁 회고 알림 처리
- 사용자 활동 로그 수집

우선순위:

- 2차 이후 완료 로그와 리포트 기능이 생긴 뒤 도입
- 알림, 메일, 자연어 파싱처럼 비동기 처리가 명확한 기능부터 적용

## 12. 개발 원칙

- 처음부터 거대한 리팩터링은 하지 않는다.
- 기존 구조를 최대한 유지하면서 점진적으로 개선한다.
- 1차 MVP를 먼저 완성한다.
- AI 기능은 후순위로 둔다.
- 사용자가 매일 들어왔을 때 Today 화면에서 바로 실행할 수 있는 구조를 우선한다.
- 완료 로그와 이월 데이터는 추후 리포트와 미룬 일 관리의 기반이 되도록 설계한다.

## 13. 모바일 앱 확장 방향

나중에는 ToDoLab을 앱으로도 사용할 수 있게 분리할 가능성을 열어둔다.
다만 지금 당장 앱을 따로 만들기보다는, 웹 MVP를 먼저 안정화하고 앱으로 분리하기 쉬운 구조를 만들어두는 쪽이 좋다.

진행 순서:

1. 현재 Spring MVC + Thymeleaf 화면으로 MVP를 완성한다.
2. API를 화면과 독립적으로 안정화한다.
3. 모바일 웹 사용성을 개선하고 PWA 가능성을 검토한다.
4. 사용 흐름이 검증되면 React Native, Flutter, Kotlin/Swift 같은 앱 분리를 검토한다.

앱까지 고려해서 지금부터 신경 쓸 부분:

- 화면 전용 로직과 API 로직을 분리한다.
- API 응답 형식을 일관되게 유지한다.
- 로그인/세션 구조는 나중에 앱에서도 쓸 수 있게 설계한다.
- 날짜/시간 처리는 서버와 클라이언트 기준을 명확히 한다.
- 푸시 알림이 필요한 기능을 미리 후보로 정리한다.
- Today, D-Day, 저녁 회고, 미룬 일 알림은 앱에서 특히 중요한 기능으로 본다.

앱에서 매력적인 기능 후보:

- 오늘 할 일 푸시 알림
- D-Day 임박 알림
- 저녁 회고 알림
- 계속 미룬 일 정리 알림
- 홈 화면 위젯 또는 잠금 화면 위젯
- 빠른 기록 바로가기

원칙:

- 당장은 웹 MVP가 우선이다.
- 앱을 이유로 현재 구조를 과하게 추상화하지 않는다.
- 대신 View에만 묶인 로직을 늘리지 않고, 핵심 흐름은 API와 Service에서 재사용 가능하게 둔다.
